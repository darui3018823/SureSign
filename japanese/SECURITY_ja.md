# セキュリティポリシー

> 📖 **Languages:** [English](../SECURITY.md) | [日本語](./SECURITY_ja.md)

## 脆弱性の報告

**セキュリティ脆弱性については、公開イシューを開かないでください。**

SureSignでセキュリティ脆弱性を発見した場合、[security@daruks.com](mailto:security@daruks.com)にメールでお知らせください。以下の情報を含めてください：

- 脆弱性の説明
- 再現手順（該当する場合）
- 潜在的な影響
- 提案された修正方法（ある場合）

以下のように対応します：
1. レポート受信確認を48時間以内に送信
2. 脆弱性を調査
3. 修正を開発してテスト
4. 可能な限り早急に修正版をリリース
5. セキュリティアドバイザリーでクレジット表記（希望しない場合は匿名）

## サポート対象バージョン

| バージョン | ステータス |
|-----------|----------|
| v1.0.x    | 積極的にサポート |

常に最新バージョンを使用されることをお勧めします。

## セキュリティベストプラクティス

SureSignを使用する際：

- Rustと依存関係を最新に保つ
- 本番環境での実行前にコードを確認
- 証明書と鍵に適切なファイルパーミッションを設定
- 暗号化キー管理についてのベストプラクティスに従う
- 不審な活動や潜在的な脆弱性について報告

## 暗号化規格

SureSignは以下を使用します：
- **RSA**: デジタル署名用（RSA-2048以上推奨）
- **AES-256**: 暗号化用（該当する場合）
- **SHA-256**: ハッシング用

すべての暗号化操作は確立されたライブラリ（rcgen、p12）を使用します。

## 依存関係

セキュリティパッチが適用されていることを確認するため、依存関係を定期的にレビューおよび更新しています。依存関係は以下で確認できます：

```bash
cargo tree
cargo audit
```

## 脆弱性開示タイムライン

責任ある開示プラクティスに従います：
- **Day 0**: 脆弱性報告
- **Day 1-7**: 初期評価と確認
- **Day 7-30**: 修正の開発とテスト
- **Day 30+**: 公開開示とリリース

タイムラインは重大度と複雑さに応じて異なる場合があります。
